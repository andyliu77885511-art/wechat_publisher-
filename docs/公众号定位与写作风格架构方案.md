# 公众号定位与写作风格架构方案

## 1. 需求概述
为 wechat_publisher 增加多定位支持：
- **财经定位**：人工精心设计 prompt，内嵌顶级财经号写作特征
- **其他定位**（科技、生活、教育、职场等）：两阶段 DeepSeek 调用，动态生成写作指令 + 生成文章
- **写作风格**：严肃/轻松/深度/故事，作为文章级配置

## 2. 技术方案设计

### 2.2 两阶段调用流程
```
1. 用户选择【定位=科技】+【风格=深度分析】
2. 系统检查缓存：
   - 缓存命中 → 直接读取缓存的写作指令
   - 缓存未命中 → 进入阶段1
3. 阶段1：生成领域指令（5-8秒）
   - 调用 DeepSeek，Prompt: "作为科技领域的公众号作者，请分析头部科技号写作特征"
   - 约束输出为结构化JSON：
     {
       "title_guidelines": [...],
       "structure_rules": [...],
       "tone_characteristics": [...],
       "forbidden_phrases": [...],
       "excellent_examples": [...]
     }
   - 存入缓存（定位+风格为key）
4. 阶段2：生成文章（15-20秒）
   - 调用 DeepSeek，组合指令：
     [领域指令] + [风格指令] + [转录稿] + [格式要求]
   - 生成最终文章
5. 更新数据库记录，标注为"科技|深度分析"定位
```

### 2.3 延迟分析（基于 DeepSeek API 基准测试）

| 阶段 | 调用次数 | 预估时间 | 备注 |
|------|----------|----------|------|
| 阶段1：指令生成 | 1 | 5-8秒 | 500-800 tokens，响应较快 |
| 阶段2：文章生成 | 1 | 15-20秒 | 1200-2000字，约2000-3000 tokens |
| **总计（首次）** | **2** | **20-28秒** | 缓存未命中的最坏情况 |
| **总计（缓存命中）** | **1** | **15-20秒** | 与当前单次调用相当 |

**延迟增加**：缓存未命中时增加10-20秒，命中时无影响。

### 2.4 缓存策略设计

#### 2.4.1 缓存方案对比
```
方案A：文件缓存（推荐MVP阶段）
├─ 路径：./cache/instructions/
├─ 文件名：{category}_{style}.json
├─ 结构：{生成时间, 指令内容, 调用次数}
└─ TTL：7天，LRU淘汰旧文件

方案B：Redis缓存（未来扩展）
├─ Key：instruction:{category}:{style}
├─ Value：JSON序列化指令
├─ TTL：604800秒（7天）
└─ 内存预估：20个定位×4风格=80个key，每个2KB，共160KB

方案C：SQLite缓存（与现有DB集成）
├─ 表：instruction_cache
├─ 字段：category, style, content, created_at, hit_count
└─ 清理：定期删除7天前记录
```

**推荐使用方案A（文件缓存）**，理由：
1. 无外部依赖，部署简单
2. 与 Streamlit Cloud 兼容性好
3. 开发成本低，可快速上线
4. 7天后自然过期，空间可控

#### 2.4.2 缓存命中率优化
- **预热策略**：部署时生成常用组合（科技|深度，生活|轻松等）
- **降级策略**：缓存失败时降级到通用 prompt
- **监控指标**：命中率、平均响应时间、缓存大小

### 2.5 质量稳定性保障

#### 2.5.1 指令生成质量约束
```
模板约束：
"请分析{定位}领域头部公众号的写作特征，输出JSON格式：
{
  "title_guidelines": ["要求1", "要求2"],
  "structure_rules": ["要求1", "要求2"],
  "tone_characteristics": ["特征1", "特征2"],
  "forbidden_phrases": ["禁忌1", "禁忌2"],
  "excellent_examples": ["示例标题1", "示例标题2"]
}"

范例约束（注入高质量示例）：
对于"科技"定位，注入：
- 参考公众号：虎嗅、36氪、爱范儿
- 优秀标题："ChatGPT火了，但AI的"iPhone时刻"还没来"
- 结构特征：痛点开头 → 技术原理 → 应用场景 → 行业影响
```

#### 2.5.2 失败重试与降级
```
三级降级策略：
1. 首次失败 → 重试1次（指数退避）
2. 重试失败 → 使用通用领域模板
3. 通用模板失败 → 使用财经模板（最稳定）

质量监控点：
- 指令长度：300-800字范围检查
- JSON格式：严格解析校验
- 禁忌词：检查是否包含"抄袭""虚构"等风险词
```

#### 2.6.2 风险点与缓解
```
风险1：DeepSeek 响应时间波动
缓解：设置超时（阶段1:10s，阶段2:30s），降级到通用模板

风险2：指令生成质量不稳定
缓解：模板强约束 + 范例注入 + 人工审核样本

风险3：缓存文件权限问题（Streamlit Cloud）
缓解：使用/tmp目录 + 定期清理 + 异常捕获

风险4：组合爆炸（20定位×4风格=80种）
缓解：LRU淘汰 + 只缓存实际使用的组合
```

## 3. 性能测试计划

### 3.1 基准测试项
```
1. 冷启动测试（无缓存）
   - 科技|深度分析：完整两阶段耗时
   - 财经|严肃专业：单阶段耗时（对照组）

2. 热缓存测试
   - 相同定位风格重复调用
   - 不同定位风格切换调用

3. 并发测试
   - 2个用户同时调用不同定位
   - 2个用户同时调用相同定位（缓存锁测试）

4. 稳定性测试
   - 连续调用50次，统计成功率
   - 模拟网络超时，测试降级机制
```

### 3.2 验收指标
```
核心指标：
- 平均响应时间：≤25秒（p95）
- 缓存命中率：≥70%（运行一周后）
- 指令生成成功率：≥95%
- 文章质量满意度：人工评估≥4/5分

扩展指标：
- 内存使用：缓存目录≤50MB
- API调用成本：两阶段比单阶段增加≤30%
- 用户切换定位延迟：≤1秒
```

## 5. 结论与建议

### 5.1 技术可行性结论
✅ **可行**，基于以下理由：
1. 延迟增加可控（10-20秒），缓存命中后可消除
2. 质量风险可通过模板约束和范例注入缓解
3. 实现复杂度中等，6小时可完成核心功能
4. 与现有架构兼容性好，无需大规模重构

### 5.2 推荐实施方案
**采用渐进式两阶段+智能缓存方案**：
1. **财经定位**：维持现有精心设计 prompt，不改变
2. **其他定位**：采用两阶段调用，首次生成指令+缓存复用
3. **缓存策略**：文件缓存（MVP）→ Redis缓存（扩展）
4. **监控**：实现基础命中率和响应时间监控

### 5.3 风险控制措施
1. **性能兜底**：设置超时，降级到通用模板
2. **质量兜底**：人工审核首批生成的指令模板
3. **运维兜底**：定期清理缓存，防止磁盘占满
4. **用户兜底**：提供"重新生成"按钮，允许手动刷新

**预估完成时间**：核心功能6小时，完整方案14小时。
